# 管理端业务控制台规划文档

> 创建时间：2026-03-25

## 一、目标定位

管理端不是通用后台，也不是单纯的 UI 原型集合，而是直播业务控制台。

核心目标：围绕“一场直播”完成创建、配置、运行监控、考勤记录、回放管理、模拟验证的完整闭环。

设计原则：
- 尽量还原现有自研管理平台的框架感
- 以直播列表为中心组织业务
- 页面要体现业务链路，不只是展示组件
- 后续老师端、学生端状态要能从管理端配置反推出来

## 二、核心业务对象

1. 机构
- 机构名称、区域、标签、启用状态
- AppKey、AppSecret、业务平台域名、接口路径

2. 直播房间
- 房间名称、房间 UUID、归属机构
- 直播类型：真实直播 / 公开课 / 预录直播 / 回放
- 进入方式：实名 / 匿名 / 白名单 / 业务平台对接
- 聊天规则：允许聊天 / 全员禁言
- 摄像头规则：是否显示老师摄像头区

3. 直播场次
- 开始时间、结束时间、课节时长
- 主讲、助教、密钥
- 当前状态：未开始 / 直播中 / 已结束 / 已转回放 / 异常结束

4. 行为记录
- 老师进入/退出时间
- 学生首次进入、最后退出、累计观看时长
- 迟到、早退、中途离开次数
- 进出教室时间轴

5. 回放片段
- 回放地址、片段名称、生成状态
- 是否对学生可见
- 是否需要补录/替换

## 三、一级模块

1. 机构与对接配置
- 机构管理
- 对接信息管理

2. 直播中心
- 直播列表
- 直播详情
- 进出教室记录
- 授课统计方式
- 直播回放列表

3. 直播监课
- 正在直播列表
- 实时状态与控制动作

4. 模拟与验证
- 模拟直播
- 老师端/学生端联调入口

## 四、关键页面定义

### 1. 直播列表
- 管理端核心主表，优先级最高
- 字段参考线上平台：机构名称、房间名称、房间 UUID、直播状态、开始时间、结束时间、课节时长、学员数、主讲姓名、主讲密钥、助教、助教密钥、标签、创建时间、回放地址
- 每条记录操作区至少包含：进出教室记录、授课统计方式、直播回放列表

### 2. 进出教室记录
- 直播业务明细页，承担考勤与行为审计
- 字段：角色、用户ID、姓名、上课总时长、进入时长占比、是否迟到、是否早退、首次进入、最后退出、中途离开次数、进出教室详情时间轴
- 顶部应包含：直播起止时间、总时长、直播状态、自动刷新、导出考勤记录

### 3. 直播回放列表
- 一场直播可能拆成多个回放片段，不是单视频
- 支持：原生回放资源、视频上传、预览视频、可见性状态、补录兜底

### 4. 直播监课
- 展示所有正在进行中的直播场次
- 支持：查看在线人数、老师状态、聊天状态、强制关闭、禁言控制、聊天开关

### 5. 机构与对接配置
- 参考线上平台的“机构管理”结构
- 上层管理对接信息，下层管理机构主数据

## 五、框架还原要求

- 保留左侧导航 + 顶部栏 + 标签页结构
- 保留“列表页 + 工具栏 + 搜索区 + 主表 + 操作列”的页面组织方式
- 保留信息密度较高的业务表格风格，不做成通用 SaaS 卡片后台
- 表格页优先强调业务字段与操作，不优先强调视觉装饰

## 六、开发顺序

第一阶段：重做管理端壳子与数据模型
- 统一导航、顶栏、标签页、Mock 数据结构

第二阶段：优先完成 3 个核心页面
- 直播列表
- 进出教室记录
- 直播回放列表

第三阶段：补全运营控制能力
- 直播监课
- 机构与对接配置
- 模拟直播

第四阶段：为三端联动做准备
- 抽共享直播状态数据
- 后续驱动老师端和学生端展示真实业务状态

## 七、验收标准

- 打开管理端后，能明显感知“直播列表是核心入口”
- 从直播列表可以进入直播详情链路，而不是孤立页面
- 进出教室记录页能够体现考勤业务价值
- 回放页能够体现分段回放与补录逻辑
- 页面风格和信息组织方式接近现有自研管理平台
